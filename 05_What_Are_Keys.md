**What are keys** 

\---

**Intro**   
	Relational databases are known for their flexibility in data modeling and the ability to store many to many relationships. The data is logically stored in relations called tables, however this is insufficient to build complex relationships. The breakthrough of the relational model is to forgo storing relationships structurally and build them at run time through the use of joins. Keys are elements which enable joins to function. By definition a key is a field (an attribute or collection thereof ) which uniquely identifies the rows within a relation. This system of identification is what allows for complex relationships to be implemented and their integrity protected. If two keys are identical then they represent the same real world entity and can thus be joined across tables within a database to form the relationships they represent.  
**Logical View**   
	Understanding the logical function of keys is one of the most fundamental relational database skills. As the introduction discussed keys are quintessential to the construction of relationships in a relational database. Having a pair of keys (primary and foreign) allows a database to make connections between tables. There are however advantages in addition to this. Primary and alternate keys enforce uniqueness. By creating such a key on a table the database is informed that elements of the key should uniquely identify rows, thus they must be unique themselves. This enables the database to perform checks when making writes. It will literally check to see if the key value already appears in the table. If it does, then the RDBMS will restrict the insert or update from being performed, raising a flag to the user. This function of keys protects the data against contradictory values by ensuring that no such situation can occur.   
A foreign key plays a different role in protecting data integrity. A foreign key will appear on a table relating it back to the primary key on another. By enforcing referential integrity the foreign key can prevent illegitimate data entries. The foreign key references back to its parent table and upon inserts or updates the RDBMS will check to see if the foreign key value appears in the primary key of the parent table. If it does not then the RDBMS will raise a flag and halt the write. The use of these key pairs also enables the database to perform cascading actions upon deletion in order to maintain integrity (for more on this see section foreign keys).   
**Physical View**   
	The physical view is how the database is able to accomplish the features from the logical perspective. The primary physical drive of these operations is the use of indexes. When a primary or alternate key is created the RDBMS will create an index behind the scene. This index is then able to be checked upon writes to ensure it is not already present in the table. Similarly when a write is made containing a foreign key the RDBMS will utilize the parents table primary key index to see if the value is present. These indexes serve a secondary role as they are able to be used by the query execution engine to build the algorithmic structure of a query (this will be discussed in more detail in a later essay).  
	We will now commence with a discussion on the categories of keys within relational databases.   
**Supper keys and Candidate keys**   
	A supper key is any group of attributes which uniquely identify rows within a table. This is not something which is created/named in a database however it is critical that every table has an eligible superkey. If there exists no supper key then the records within a table are indistinguishable,  making the table of little value analytically.   
A candidate key is a minimal subset of the superkeys attributes which uniquely identify records. The name candidate stems from the key's eligibility to become a primary key. A table can have multiple candidate keys. This occurs when there are multiple distinct subsets of the superkey which yield valid record identifiers. Candidate keys are not named or constructed within the database, however they are necessary to identify. These keys will later be named as primary or alternate keys, which are critical to the operation of a database. Special care must be taken in the identification of such keys as it must be guaranteed that they will always be unique and are not just unique for the current subset of records being observed.   
**Primary keys**   
	Primary keys are the bedrock of the relational database. There can only be one primary key per table. This key is chosen out of those candidate keys which have been identified. These are the references to which all children concerning that table (as a parent) will be in reference to. Once declared the primary key will enforce uniqueness and not null constraints upon the entries of its attributes. The importance of uniqueness has already been discussed, but a word on not null is necessary. The primary key of a table can not contain null values because that would serve to invalidate its unique constraint. Null values are considered unequal to any other values including other nulls within a relational database. This would pose a significant problem when ensuring uniqueness, as many null values could be added to the primary key columns. This would then make referencing them from other tables impossible as there would be no equal records. There are some instances where a null value should in fact be recorded as a unique event within a database. The most common place for this need is in data warehousing when companies want to conduct analytics. The way around this is to use a surrogate key(see section natural vs surrogate keys) to identify some value as ‘being null’ thus analysis can be conducted on all those records which contain this null surrogate key (more on this subject as it related to data warehousing in a later essay).   
	As primary keys are so critical to the integrity of data and the formation of valid relationships amongst tables within a database it is recommended that every table should contain one. The choice of what columns should be contained within the primary key is critical to the performance of a database. They should however, follow a general rule of durability. Durability is the key's ability to resist change. There are many business reasons why keys may change over time but a durable key will be resistant to these changes. Many of the topics below are variants of primary keys which are given their own name.  
**Composite keys**   
	A composite key is a key that is made up of more than one attribute. Composite keys are necessary when no single attribute is unique within the table, but uniqueness is present across a subset of attributes.  An example of such a case could be an enrollment table at a university where (student\_id, course\_id) could be a primary key. student \_id will not be sufficient because the same student will be taking multiple courses. Similarly many students will be simultaneously taking a course thus course\_id is also insufficient. The composite (student\_id, course\_id) will however describe the table well.   
	One downside of a composite key is the index enforcement. Compared to a single column key the index will be wide; containing multiple attributes (this too is known as a composite index). This has an effect on query performance as well. A composite index will be ordered by the first (left most) attribute. The ordering of the other columns only plays a role if the ordering of the previous (one to the left) are equal. Using the previous example (U1234, bio\_2260) would come before (U3241, astro\_1800) but if the student is identical the ordering will fall to the second attribute. With more attributes this process will continue to cascade. This means that queries filtering on the key columns will only be efficient for the left most column. The ordering of the composite keys attributes is therefore important to performance (more on composite indexes in a later essay).   
**Natural vs Surrogate keys**   
	A natural key is both readable and meaningful to humans. These are often developed and dependent on business logic. An example would be the name or abbreviation of a department within a business. Its meaningfulness is its main strength. On the other hand being tied to business logic means it is highly susceptible to change. This is a major disadvantage for a key as discussed earlier. If a key is changed it will require a cascading update of all child tables. This means every row in every table the key is present must be updated to reflect its new value. The process can be slow and dangerous as the large number of updates gives opportunity for an anomaly to occur. Its human readability is also a disadvantage as this will often require a lot of space (a large datatype). Having large datatypes increases index bloat which is detrimental to performance.   
	Directly opposed to this is a surrogate key. Surrogate keys are artificial identifiers; they have no human meaning. The most common way to store surrogate keys is by an incrementing integer. This can vastly decrease the size of the data making for smaller faster indexes. An additional benefit to this is the keys' durability. The business logic which the key is representing can have a name change but the surrogate key will be unaffected. This is a massive advantage over a natural key as it avoids a cascading update.   
	The durability and light weight of a surrogate key often lead to it being preferred over a natural key. This is especially true in data warehousing where data has an especially long life span. This lifespan leaves it vulnerable to changes in business logic. The vast amount of data within the fact table of a database means large data types would create massive index bloating(more on data warehousing and fact tables in a later essay). The surrogate keys allows for you to keep the natural human readable column within the table as an alternate key (see section alternate keys for more information), therefore filtering can be performed on that column. While having the surrogate key present in its own column which is used for the background logic of query execution. This combined approach allows for increased performance with little to no loss is database navigation easy.   
**Intelligent keys**   
	An intelligent key is one which conveys meaning in its design. These are typically a subtype of natural keys as they are created and used in business logic. An intelligent key can be well exemplified with a product inventory example. Let a department store have a database tracking the products in their inventory. An example of an intelligent key to track a television could be ELEC-TV-55-2024. This key is intelligent because it contains descriptive information within it. ELEC describes that the item is held within the electronics department, TV is the type of item, 55 is the size of the television and 2024 is its model date.    
At first consideration this seems like a smart use of a key as it can contain a large amount of information on the item. Due to the nature of indexes however, it is not very useful. Because indexes are sorted only the first segment of the key can be queried for efficiently. Furthermore you must sort this information within the index which will lead to index bloat. The combination of these two effects makes the intelligent key not as effective as one may think.Key durability must also be considered. If at some time the store decides to change the department in which they sell the television or the product identification code TV a forced cascading update will core.   
For these reasons intelligent keys are generally discouraged within databases. The information within them is however useful. The elements of an intelligent key often make good attributes for the table as they hold significant business logic. Thus intelligent keys are often split into their elementary parts and stored within the attributes of a row. This gives the added advantage of being able to build indexes on the individual components and improve query performance.   
**Foreign keys**   
	A foreign key is a named attribute or subset therefore with a link in reference to a primary key present on a different table. Unlike primary keys, an unlimited number of foreign keys are permissible per table. This is the method by which a relational database creates many to many relations in runtime. By declaring a foreign key in reference to a primary key a correspondence is created between the two tables. This correspondence enforces a relationship between the tables which as explained above (see *Logical View*) enforces data integration. This relational enforcement is necessary to prevent orphaned data. Orphaned data is when there exists a child entry which can not be traced back to any entry within the parent table.  
	An additional benefit to using foreign keys is that they support cascading actions. A cascading action occurs when an element of the parent table is deleted. Such an event can trigger a ‘casaced’ across the child table where all relevant rows are altered to be either deleted or set to null based on the preference of the database designer. This helps to automatically maintain data integrity throughout the database.   
**Alternate keys**   
	As discussed there can be only one primary key declared per table. As we have seen there may be cases in which the primary key is a strict subset of the supper key. This leads to a situation where although the primary key is chosen as a reference to other tables there may be multiple columns or groups of columns which behave to uniquely identify records within the table. In this case it is valid to desire the same protective measures allotted by the primary key for these groups of columns as well. Fortunately an alternate key fills this role. There is no upper limit to the number of alternate keys on a table, and they share all the protective measures as a primary key without being used to reference foreign keys. Alternate keys are evoked by using a UNIQUE constraint (more on SQl syntax in a later essay). Alternate keys are also allowed (in many RDBMS) to be referenced by a foreign key thus allowing for the same protective measures on their child tables.  
Alternate keys are particularly useful when utilizing surrogate keys. Because surrogate keys are meant to identify a candidate key typically by an incrementing integer a table with a surrogate key will have columns which also uniquely identify the records but are larger or less durable than the surrogate. If the uniqueness of these columns is broken then the uniqueness of the surrogate key will also be technically invalidated. This is known as a contradiction- when different keys reference the same date. Thus by declaring these columns as an alternate key they can be protected against duplication and null values, preventing the assignment of contradicting surrogate keys. Therefore greatly increases data integrity at the cost of index space and write time. It is however probable that queries will frequently filter on these columns as they for the business logic of the table thus an index is often useful outside of its protective advantages.  
**Summary**   
	Keys are an essential aspect of relational databases. They provide identity to relational records, enable the construction of relationships through joins, and form the backbone of relational integrity. This is accomplished through the use of indexing which can have benefits to query performance at the cost of write speed. By implementing keys, update and insert anomalies, contradictory data, orphaned records, and inconsistent relationships can help to be avoided. As seen in *Essay 4 Normalization* these are critical aspects in protecting a database and establishing its schema.
