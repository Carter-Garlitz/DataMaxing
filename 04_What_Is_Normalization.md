**What Is Normalization**  
\---

**Intro**   
	Now that the basics of a relational database are understood it is time to discuss the details of its layout. The previous essay hinted at the idea of normalization and this one shall cover it and its forms in more detail. Understanding the different forms and the pros and cons thereof is critical for database design and performance.   
In general the higher the normal form of a database the less duplicated data there will be thus saving space. This will require the use of more keys which will also help to ensure data integrity and relationship preservation. Higher normal forms also help to improve write and update performance in two major ways. Because data is isolated to specific tables, when updating values you need only update them in one place, then the associated foreign keys present in other tables will relate back to the correct information. For example if a hospital has a table exclusively for patient information and someone changes their address, it is only necessary to update that one row in order to ensure this information is updated correctly. Because all relevant tables will link to this one through foreign keys any query will now reflect the updated information. On the other hand if the database was denormalized they would need to update this information in many rows across multiple tables, increasing the update time and probability of data anomalies. A data anomaly is when data is updated, inserted or deleted incorrectly. The second reason for improvement is due to decreased locking when writes occur. Because the write will take less time the lock will be shorter in duration, and because it will only affect one row on one table the scope of the lock will be greatly decreased from the de-normalized equivalent.   
Normalization for all its benefits, does however come at some cost. The more isolated the data the more keys will be needed in order to form and retain relationships amongst tables. The maintenance of keys requires a fair amount of overhead concerning both I/O and cpu because a key requires the creation and maintenance of an index. When there are many keys this can accumulate significant resource allocation. Additionally because the data is stored in separate tables, when queerying there will by necessity be more joins. This uses a lot of cpu and memory resources increasing the time and cost of queries. Finally, the segmentation of data across many tables can make the database more difficult for people to understand. This becomes more of a problem when people need to access the database who may not be an expert in sql or may not interact with the database at high frequency such as business intelligence personnel and stakeholders.   
Due to these pros and cons it can be harmful to pursue high levels of normalization in certain databases. The needs of a database will determine the level of normalization of the data. For example a data warehouse, which is used for analytics and stores vast amounts of data the majority of which being located in the fact table. This means it will have high querying demand and must be able to support complex querying. Because of this they are often left in a fairly denormalized state compared to transaction databases as normalizing dimension tables (tables which store details about the facts) will have a small effect on the overall size of the database and the avoidance of complex joins proves to be more beneficial. This has the added benefit of making the database more comprehensible for business analysis to perform ad-hoc queries. The star schema which is common within data warehousing is a schema specifically designed to be relatively denormalized thus increasing query performance (more will be said on data warehousing and the star scheme in later essays). Fortunately there are multiple levels of normalization that can easily be incorporated into a database. A discussion on the different level is well warranted.   
**NF0**  
	Normal form zero does not get deployed in a database. There are no constraints on normal for zero. There can be columns with mismatched datatypes, repeated groups, no keys or row identifiers. Many of the mistakes this form permits will not even be allowed by the database itself. This makes normal NF0 more of a theoretical starting point then something to be encountered in real life. Anything falling short of normal form one would be in this category but should be improved such that it can be moved up into a true normalized form.   
**NF1**  
	The first normal form is the minimum level of normalization. Every relational database no matter its function should qualify as at least NF1 as it enables the database to function in the way it is designed. The first rule is that row order should not be used to convey meaning. As relations are defined as unordered sets encoding meaning into the order of records is not robust and could change at any time. Second, there should be no columns which contain a mix of data types. This is easy to enforce as a RDBMS will not even allow for this to occur. Thirdly, repeating groups are not allowed. This means that each column should contain a single indivisible element of data. A counter example could be a column titled contact\_info which contains an email phone number and website address separated by commas. This information should be split into three distinct columns; phone\_number, email, and website. Having multiple elements within a column can make querying difficult as indexing will only order by the first element within the column making the others difficult to find. Additionally some records may not have a website and thus attempting to parse the data in a search by position could also become invalid. It is important to know with certainty what data will be in each column. With repeating groups this is made volatile. Lastly, each table should at least contain a candidate key (this is a set of attributes which serve to uniquely identify records), though it is most common and beneficial for a primary key to be explicitly named(see essay 5 *What Are Keys* for more information). This ensures a unique row identity and protects against repeating contradictory values (with RDBMS enforcement only if a primary key is named). For example with no primary key you could have a situation in an employee database where the same employee\_id shows up twice each referring to a different name. With primary key enforcement on employee\_id this will not be allowed by the database.     
**NF2**  
	Second normal form builds on top of NF1 with two additional rules. The first is there must be a primary key. This is a somewhat trivial addition to NF1 as in most cases this will be satisfied by NF1. The second rule is that each non-key attribute must be dependent on the entire key. This means that if a column is not included in the key it must be uniquely identified by the combination of every column within the key. This helps protect the data against update, delete and insert anomalies. As a counter example consider a department store inventory table. Let the table consist of the attributes: department, item, count, manager with (department, item) as the primary key. Clearly the count depends on both department and item (the same item may be sold in different departments), but manager only depends on the department and thus breaks NF2. This could lead to many different problems. Say the inventory of a department is empty and thus has no rows. This would leave you with no knowledge of who the manager is (an delete anomaly). If the manager of a department is changed then you must go through every item in that department and change the associated manager which increases the probability of an error occurring (update anomaly). When inserting into the table you must always enter the manager in addition to the other information. This could lead to a data entry error resulting in a contradiction amongst the rows (insert anomaly). Because manager depends only on the department it would be beneficial to remove that attribute from the inventory table in order to create a new management table with department as the primary key and manager as a non-key attribute.   
**NF3**  
The third normal form builds upon the second. This is the most common level of normalization used in databases. The constraint that identifies NF3 is that transitive dependencies are not allowed. That is to say that non-key attributes should not depend upon other non-key attributes. A common slogan for NF3 is ‘the key, the whole key and nothing but the key’ which well exemplifies this rule. As an example let there be a table showing student information in a college. This table contains the attributes; student\_id (primary key), last\_name, department\_id(their field of study) , department\_name, department\_chairman. Here each attribute depends on the primary key and the whole primary key (trivially as it is only one column). There is a dependency however on department\_name, department\_id and  department\_chairman, none of which are the primary key thus violating NF3. This would expose the table to anomaly risks. It would be better to have a department table with department\_id, department\_name and chairman therefore upon any change there would only be one place that would require changing.   
**BCNF**  
	Boyce Codd Normal Form or BCNF is a bit of an honorable mention. It is almost identical to NF3 with only a slight difference in constraint. This being that, every attribute must depend on the entire key and nothing else. This places the constraint onto every attribute in the table, not just the non-key columns. This means in addition to NF3  there can not be a dependency amongst any columns within the key. This is a generally stronger form of the rule and should be applied in most cases when NF3 is being applied. For example, suppose the department table from the previous example had \[department, department\_name\] as a primary key. This would in fact satisfy NF3 as the chairman depends on the whole primary key. However, a dependency on the key attributes would violate Boyce Codd Normal Form. If the department was the primary key here then the table satisfies BCNF.   
**NF4**  
	The fourth normal form is useful when there is a multi-valued dependency between data. A multi-valued dependency occurs when there are at least three attributes X, Y and Z, where X determines the set of values of Y and Z while Y and Z are independent from each other. By separating these dependencies into their own tables (one for x and Y and another for X and Z) anomaly risk can be minimized. As always the best way to explain this is through an example. Let there be a university that is trying to maintain a database of students, the courses they take and the clubs they are a part of. This could be stored in a single table with columns student\_id, course, club. Because course and club are dependent on the student\_id but independent of each other this is a mulit-vlued dependency and should be separated. If a student leaves the university they must be deleted from multiple rows. If they join a new club there must be multiple rows inserted. The violation of NF4 exposes the table to anomaly risks. By splitting the table in two- one table containing student\_id and course and one containing student \_ id and club- the database can be put into NF4. This allows for only a unique combination of the two columns to be stored in each table. This will save space and reduce the chances of anomalies occurring.   
**NF5**  
	The Fifth normal form is the highest level of normalization in classical literature and is less common to come across in practice. It differs from the other forms as it relates to join dependencies instead of key dependencies. At this level tables cannot be the result of a logical join from other tables. In essence ‘if it can be its own table it should be’. In the contrapositive, it can be said that any table cannot be broken down into smaller tables without losing data. A classic example is the supplier, part, project relationship. This is where parts are given by suppliers, parts are used on projects and suppliers can work on projects, but no set of two relationships imply the third. Eg. a project can use a part, one supplier can deliver the part and a second supplier can work on the project. The only way to represent this relationship without exposing it to unnecessary anomalies is NF5. resulting in the construction of three tables; (supplier, part), (part, project), (supplier, project).   
**Summary**    
	There exists many different relationships between data and purposes of the databases which hold them. Normalization is a design choice in order to optimize the database for the function it was intended for. Each normal form addresses different structures and relationships amongst data while protecting against anomalies. There is always a trade off when increasing the level of normalization. To reduce redundancy you must increase joins, and to improve read and write speed is to increase key overhead and query complexity. At the heart of normalization are keys and dependencies. The next essay will explore this in greater detail.
