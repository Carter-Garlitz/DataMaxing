 **What Is a Relational Database** 

\---

**Intro**   
	Due to their clear structure, consistency guarantees, and general flexibility relational models dominate industries like business, finance, and enterprise systems. Being the most widely used for creating data applications it is crucial to understand how these systems operate. A deep understanding of the operations will help to improve the efficiency and effectiveness of applications. This essay marks the beginning of a deep dive into relational models, focusing on the fundamentals of their design and construction.   
**Physical Storage**   
	The relational database system or RDBMS acts as a layer of abstraction between the user and the storage and retrieval of data. Although it is not common to work with the underlying physical storage and operation of the RDBMS, having an understanding of how it works will provide intuition for why the logical layer works the way it does.   
	The RDBMS writes data to disk and will pull this data into memory when it needs to make reads, writes or updates. The data is written to disk on fixed sized blocks (typically 8 or 16 kb) called pages. This is the smallest unit of IO the database can move between disk and memory. These pages are stored in database files depending on the kind of data they hold. File types include data files, index files, system catalog files, and log files.   
By default tables are laid out as heaps on disk. A head is an unordered set of data, thus the row ordering is not guaranteed. By creating a clustered index on a table the default behavior is changed. A clustered index will store the table data in a B+ tree where the rows are sorted by the key's value (more in keys in the keys and joins section and details in essay 5 ). Now when querying for data constrained on this key, the RDBMS query engine has the ability to know on which pages the data is located. This knowledge can help limit the number of pages the engine must pull into memory to read (more on how and when this occurs in a future essay). Creating an index on a column will create index file pages. These pages will consist of the sorted values of the column and a pointer to the associated row in the heap or clustered index. This will also improve the speed of queries filtered on that column. Additionally creating an index on a subset of columns will create a statistics page on the column containing information about the carnality and frequency of key values in those columns. This will help the query engine to determine the proper algorithms to retrieve the data. This will be discussed in greater detail in a later essay. For now it is clear that understanding how the system works behind the scenes will give intuition on how to optimize a database.   
Creating indexes is however not free from downsides. Every time the database is changed, like an update, write or delete, the RDBMS must also update every index defined on that table. This comes at the cost of cpu and disk I/O which are required for these operations. Therefore the more indexes you have the faster your queries may be but it will be at the cost of write performance and storage space.

**Logical View**  
The way a user of the RDBMS sees the data within it is called the logical view. Relations databases are laid out logically in the form of relations (tables). A relation is an unordered set of records (rows). Each record is a unique instance of data which captures some set of information about the real world system being recorded. A record consists of elements called attributes. Attributes form the columns of a table, and are where the database will enforce schema on write. This is done by telling the database what data type the attribute will be upon its creation. In addition to data types attributes can have constraints including UNIQUE, NOT NULL or CHECK (which is followed by a statement the data must make true)(more on these functions in a future essay). If any of these qualities are broken upon insertion or updating the database will return a flag and disallow the write to proceed.   
**ACID**  
	A database is required to be consistent. It accomplishes this by ensuring changes to data occur in a reliable and controlled manner. By grouping one or more operations into a single logical unit called a transaction, the RBDMS is able to ‘fill or kill’ the change. This makes sure that either the entire transaction is complete or none of it is. The guarantees provided by the transaction process go by the acronym ACID.   
*Atomicity*   
	A transaction is completed in full or no change is made. If any part of the transaction fails every part of the transaction is rolled back, restoring the database to the state it was in prior to the transaction.   
*Consistency*   
	A transaction must result in a valid state of the database. This includes respecting and preserving all defined rules like constraints, keys, and relationships.   
*Isolation*   
	Concurrent transactions must not interfere with each other. Each transaction is isolated and appears as if it were acting alone.   
*Durability*   
	After a transaction is committed the changes made will be persistent. This must include cases like crashes and power failures. This is accomplished through the use of write-ahead logs and recovery protocols.   
	In physical reality these ACID guarantees are accomplished by the use of write-ahead logs (wal), locking, and multi-version concurrency control (MVCC). A write-ahead log ensures atomicity and durability by writing the transaction to a log and flushing that log to disk before it makes any changes to the database. This ensures that if something goes wrong during a transaction and it is partially completed, the database has the information to either complete or rollback the particle finished transaction. Locks ensure that two conflicting requests to the database will not be simultaneously executed. For example it would stop two people from simultaneously updating the same row. Locking can occur on a row, page or a table. This comes at the cost of reduced concurrency. MVCC helps to reduce the concurrency issue caused by locking. It works by allowing transactions to operate on two different versions of the same table, old and new. When a system accesses the database its transactions will operate on the version of the table present when their system started. Any changes made to the database by others will not appear to them until they restart their system. This enables readers and writers to access the table at the same time. All of these processes work together to ensure ACID compliance.   
**Keys and Joins**   
	Keys and joins are inherent to the functionality of a relational database. Relationships within a relational database are not held explicitly. The only explicit relationship is between the attributes in each record. Keys and joins are used together to build relationships at query time. By writing commands in structured query language or SQL, the user is able to combine the attributes from multiple tables to form a join table which expresses the relationships of the data. This process allows for data to be stored in a single place within the database. For example there can be one department table which contains the department name, id, supervisor id and location and another separate employee table containing the employee names, and department id. Now it is possible to find the location and manager name of each employee by combining the two tables where they share a department id, thus there is no need to store all the information regarding the department redundantly within each employee record. This is an example of normalization which is a critical aspect of relational database design and will be discussed in detail in the future(see essay 4).   
	Keys serve the function of identifying rows in a table. There are many types of keys in a relational database but they all serve a similar purpose. They are the objects on which the combination of tables can be made. They function by being present on two tables. On the one in which they are used to uniquely identify rows they are called primary keys (there can only be one primary key per table) . On the other table each can can appear as often as necessary, these keys are called foreign keys and directly link back to the primary key table. This linkage between the tables is enforced by the RDBMS. This means it will not allow for a foreign key to exist that does not link back to a primary key. This helps the database maintain integrity during writes and updates.   
	Joins are the operations which combine tables by their shared key(it is useful here to note that joins do not require the existence of keys, and can be used on attributes not named as keys, though this may come at the cost of decreased integrity and increased time computation and memory requirements). Thus joins are one of if not the most useful features of a relational database. Without joins a RDBMS would be unable to effectively model many to many relationships. There are a few different types of joins available within the database. Each join gives the user different ways in which to combine tables. Furthermore because SQL is a declarative language behind the seance there are a few different algorithms which the SQL engine can deploy in order to perform the joins effectively. Both the types of joins and the algorithms used to execute them are rich topics of discussion which fall outside the scope of this paper. They will have their own dedicated papers to discuss them properly.    
**Summary**   
	This essay has begun the deeper discussion of the intricacies of a relational database and the functions of a RDBMS. The RDBMS provides a layer of abstraction between the user and the physical storage of data and execution of queries. This being said, without having an understanding of what is actually occurring behind the scene developing and maintaining an effective database will be impossible. There were many topics within this essay that will be brought up repeatedly in the future and warrant entire papers in and of themselves- the following essays will be doing just that.
