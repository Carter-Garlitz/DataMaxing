**What Is a Database**

**Introduction**  
There are many needs for a program or business entity to store data which persists longer than the operations run time. Clear examples of which include; orders placed over an online transaction as it will be necessary for the business to be able to re-access this information at a future date for quality control, accounting, and many other reasons, salaries paid to employees for auditing, payment discrepancies and so on, connections between people within a social networking app as whenever someone makes a post or update they must be able to recall this information in order to display it to the correct people, the need for this type of system is endless. At a basic level databases provide a medium to store and locate such information through providing a layer of abstraction between the logical view (what you see and interact with) and physical storage (how the data is stored on disk) of data. It would also be possible to store the same information as simple files however there are many benefits a database in proper provides over a simple file storage mechanism. Understanding these advantages is a great way to learn about and understand databases.   
**Accessing data**   
Along with storage, accessing data is the main problem a database seeks to solve. All the information stored would be of no benefit if it were impossible or unrealistic to target and output specific records within the dataset. Fortunately databases solve this problem by providing sophisticated instrumentation to query (or search) for such records.   
Within a company or program there is often the need for multiple different entities to have access to the same data. A file can only exist in one place therefore if multiple entities need access to the information therein they must contain a copy of the data on their own systems. If the dataset is small this may not be an issue however most modern companies and applications rely on massive datasets up to petabytes in scale, therefore the cost of maintaining distinct copies on every system which requires access to it would be massive. Databases solve this problem by providing a centralized space to store information and giving access to that information through a shared network to any system authorized to access it allowing for only one central dataset must be stored and maintained. Additionally databases contain software which allows for concurrency (ie. simultaneous retrieval or editing) with multiple threads, this allows people to freely access the dataset when they need it. They must not wait in line to access the information.   
Allowing the same records to be duplicated and stored in decentralized spaces would pose another issue; How would you ensure that the data is accurate and consistent? Upon making the copy they would be identical however as each entity would have free access to do what they please with it, how could you ensure it remains the same. Each entity could, update, or delete as they please leading overtime to a noise mess of inconsistent data. Clearly keeping the data in a single centralized space where all actions affect the data for all entities would solve this problem, however there is still a slightly less obvious issue which may arise. How do you ensure new records are consistently formatted, so that systems know what exactly they will receive from a query.  Databases help to enforce a regularity amongst the information gathered from potentially different entities and at different times through the use of scheme enforcement. Schema is the high level design of a database; it contains meta-data which describes the attributes (constraints of the individual data entries, think columns of a table as an example) of the dataset as well as the type of object held within it. This helps to ensure consistency throughout the data so that everyone can understand what is held within it and knows what to expect from their queries. All business and applications evolve over time to fit the needs and objectives of their business logic. This can lead to an evolution of the data set itself. Maintaining a record of historical schema designs can help to track the evolution of the data set, as well as provide a way for applications to maintain compatibility with previous versions.   
**Building Relationships.**   
There exists a third and substantial problem with a simple file storage system. The world is full of connections and relationships and so is data. We have seen from the example given in the introduction that there can exist many different kinds of relationships between data. In the case of the online order each record likely contains information about what the item is along with the user that ordered it, the distribution center from which it was ordered, the payment type of the customer and so on. Furthermore the business likely stores data regarding each of these attributes, e.g. the distributor likely has information about product inventory and which employees are responsible for delivering the item. The payroll example likely has many similar relationships, it may keep track of hours worked, the days on which they worked, their hourly wage etc. furthermore each employee could belong to a department, each department to a store and aggregate information must be accessed for accounting purposes. The social media example contains a whole host of relations, each person has many friends and each of their friends many friends. If a person makes a comment on a post it must be visible to all people who follow the original poster.   
As you can see relationships among data can create highly complex structures, a file system would be bogged down by so much redundancy if it even had hope of storing all this information. A database’s aim is to solve these problems. Over time many different databases have been designed and optimized for storing data under different data models. These databases provide efficient mechanisms to store and query data under specific models making querying and storing complex data affordable and efficient.   
**Summary**   
As the world of data and analytics evolved, data stores and accessibility did alongside. Databases were designed to solve the problems of storing, retrieving and modeling behavior of data. Giving rise to the definition: *A database is a software system that manages persistent data, providing structured access, shared use, and guarantees about correctness and performance*.There are many different kinds of databases each optimized for the many different relationships that exist between real world data. Choosing one over the other is a question of tradeoffs. In the next essay we will focus on these different relationships and the trades of corresponding databases and how they achieve their performance goals.
